\section{System Overview}
The system processes a job description and a candidate pool and produces two outputs per job:
\begin{enumerate}
    \item \textbf{Rank output:} a scored, sorted list of candidates.
    \item \textbf{Insights output:} reasoning, skill gaps, and recommended next steps.
\end{enumerate}
It exposes a CLI with a strategy flag (deterministic, hybrid, llm-rag) and uses a shared schema layer to keep outputs consistent across approaches.

\section{Skill Normalization Logic}
Normalization is applied to both candidate skills and job requirements. It provides consistent matching by removing surface-level variation in user-entered skills.

\subsection{Normalization Steps}
\begin{enumerate}
    \item Trim whitespace.
    \item Lowercase all tokens.
    \item Map aliases to canonical skills using \texttt{skill\_aliases.json}.
    \item Remove empty strings.
    \item Deduplicate and sort.
\end{enumerate}

\subsection{Example}
Input: \texttt{[" React.js ", "JS", "javascript", " "]} \\ 
Output: \texttt{["javascript", "react"]}

\section{Approach 1: Deterministic}
The deterministic engine uses set operations and explicitly defined weights to compute a 100-point score. It is optimized for speed, transparency, and predictable behavior.

\subsection{Scoring Formula (Weights)}
Weights are taken from settings:
\begin{itemize}
    \item Must-have skills: 50
    \item Nice-to-have skills: 20
    \item Experience fit: 25
    \item Location fit: 5
    \item Experience bonus: up to +5 (2 points per year, capped)
\end{itemize}

\subsection{Deterministic Score}
Let:
\begin{itemize}
    \item $M$: number of must-have skills required.
    \item $m_{exp}$: explicit must-have matches.
    \item $m_{inf}$: must-have matches inferred from text.
    \item $N$: number of nice-to-have skills required.
    \item $n_{exp}$: explicit nice-to-have matches.
    \item $n_{inf}$: nice-to-have matches inferred from text.
\end{itemize}

\noindent Must-have ratio:
\[
R_{must} = \frac{m_{exp} + 0.5\,m_{inf}}{M} \quad (M>0)
\]
\noindent Nice-to-have ratio:
\[
R_{nice} = \frac{n_{exp} + 0.5\,n_{inf}}{N} \quad (N>0)
\]

\noindent Score components:
\[
S_{must} = 50 \times R_{must}
\]
\[
S_{nice} = 20 \times R_{nice}
\]
\[
S_{exp} =
\begin{cases}
25 \times (\frac{y}{y_{req}})^2 & y < y_{req} \\
25 + \min(2(y - y_{req}), 5) & y \ge y_{req}
\end{cases}
\]
\[
S_{loc} \in \{0, 2.5, 5\} \text{ depending on remote/match/relocation}
\]

\noindent Penalty multiplier if must-haves are missing:
\[
S_{final} = (S_{must}+S_{nice}+S_{exp}+S_{loc}) \times R_{must}
\]

\noindent Status penalty (availability intent):
\[
S_{final} = S_{final} \times \begin{cases}
0.50 & \text{not looking} \\
0.90 & \text{open to offers / interviewing} \\
1.00 & \text{otherwise}
\end{cases}
\]

\subsection{Output Snippet (Deterministic)}
\begin{lstlisting}
{
  "candidateId": "c-001",
  "score": 45.56,
  "matchedSkills": ["python", "sql"],
  "missingMustHaveSkills": [],
  "missingNiceToHaveSkills": ["aws"],
  "summary": "Potential match. Meets experience requirements. Matches all core must-have requirements. (1 missing skill(s) were rescued by scanning experience/projects)."
}
\end{lstlisting}

\section{Approach 2: Hybrid}
The hybrid engine blends deterministic scoring with semantic similarity from embeddings. It is designed to reduce false negatives caused by keyword gaps and to capture semantic similarity across contexts.

\subsection{Embedding Pipeline}
\begin{itemize}
    \item Model: \texttt{BAAI/bge-small-en-v1.5}
    \item Storage: \texttt{ChromaDB} (persistent)
    \item Similarity: cosine similarity $\in [0, 1]$
\end{itemize}

\subsection{Hybrid Score Formula}
\[
S_{hybrid} = (W_{det} \times S_{det}) + (W_{sem} \times (100 \times \text{cosine}))
\]
Default weights are $W_{det}=1.0$, $W_{sem}=0.0$, and are overridden when the hybrid approach is selected.

\subsection{Output Snippet (Hybrid)}
\begin{lstlisting}
{
  "candidateId": "c-002",
  "score": 73.24,
  "reasons": [
    {"type": "semantic_similarity", "weight": 18.0}
  ]
}
\end{lstlisting}

\section{Approach 3: LLM + RAG Ranking}
The LLM + RAG engine uses a hybrid retriever and an LLM re-ranker to produce structured insights and refined scores. It is optimized for maximum semantic accuracy and human-like reasoning.

\subsection{Two-Phase Flow}
\begin{enumerate}
    \item \textbf{Hybrid Top-K:} A fast hybrid pre-filter selects the top candidates.
    \item \textbf{LLM Re-Rank:} The LLM evaluates each candidate, enforcing the same scoring constraints and schema validation via Pydantic.
\end{enumerate}

\subsection{Output Snippet (LLM + RAG)}
\begin{lstlisting}
{
  "candidateId": "c-003",
  "score": 88.5,
  "insights": {
    "strengths": ["Strong foundational alignment in backend stack."],
    "skillGaps": {"mustHave": [], "niceToHave": ["kafka"]},
    "recommendedNextSteps": ["Upskill Opportunity (KAFKA): Build a small event-driven pipeline."]
  }
}
\end{lstlisting}

\section{Trade-Offs}
\begin{itemize}
    \item \textbf{Deterministic:} Fast and transparent, but brittle to synonyms and context.
    \item \textbf{Hybrid:} Better recall and semantic matching, but adds embedding cost and complexity.
    \item \textbf{LLM + RAG:} Highest quality insights and reasoning, but slower and costlier with API dependency.
\end{itemize}

\section{Next Improvements}
\begin{itemize}
  \item Cache LLM results.
  \item Optimize the system prompt.
  \item Let the LLM self-reoptimize weights per job.
\end{itemize}

\section{Time Spent Estimate}

    \item Hybrid approach: 1.5 hours
    \item LLM + RAG approach: 1.5 hours
\end{itemize}
